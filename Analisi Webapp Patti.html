01 Riepilogo esecutivo

La webapp è funzionalmente solida e visivamente coerente — l'header istituzionale a 4 livelli, la smart topbar con filtri e URL routing, il pannello statistiche con donut chart e KPI card mostrano un lavoro di buona qualità. Tuttavia, anni di sviluppo incrementale hanno lasciato un debito tecnico significativo: due librerie di icone caricate insieme, Tailwind caricato per poche utility class, un file JS principale da 3 371 righe, e soprattutto header e footer duplicati su tutte le 8 pagine. I problemi più urgenti sono di performance (pagina pesante a causa di CSS inutilizzato) e manutenibilità (ogni modifica ai contatti richiede di editare 8 file separati).

I 3 interventi più urgenti
Massimo impatto, minimo rischio
#InterventoImpattoStima
1 Rimuovere Tailwind CSS dal index.html e sostituire le poche utility usate con classi custom −2,9 MB CSS, caricamento significativamente più veloce 2–3h
2 Unificare le icone su Font Awesome (già installato) ed eliminare Lucide + script di migrazione −60 KB JS, eliminazione di codice fragile e console clutter 3–4h
3 Estrarre header e footer in un include JS (components.js) caricato su ogni pagina Elimina 700+ righe di HTML duplicato, manutenzione centralizzata 4–6h
02 Architettura & Codice 8 issue
A1 — Header e footer duplicati su tutte le 8 pagine
Ogni pagina HTML contiene l'intero blocco header (4 livelli, ~100 righe) e footer (5 colonne, ~130 righe) copiato identico
critico
Se cambia un recapito telefonico, il nome di un funzionario o un link social, va editato a mano in tutti e 8 i file. Già oggi il footer contiene un typo ("Rigenerazione Urbana") che compare su tutte le pagine perché copiato identico.
Problema~230 righe HTML replicate × 8 file = 1 840 righe di codice ridondante. Ogni modifica richiede 8 edit sincronizzati.
SoluzioneCreare components.js con funzioni renderHeader() e renderFooter(). Ogni pagina include lo script e chiama le funzioni in 2 righe.
A2 — Due librerie di icone caricate insieme (Font Awesome + Lucide)
lucide_to_fontawesome_migration.js converte a runtime i data-lucide in classi FA
critico
L'app carica sia Font Awesome 6.5.1 (~400 KB CSS) sia Lucide via script di migrazione. Il file lucide_to_fontawesome_migration.js rimappa ogni icona Lucide in un'icona FA equivalente a runtime — un bridge fragile che deve essere aggiornato ogni volta che si aggiunge una nuova icona. Nella navbar le icone Dashboard usano fas fa-map-marked-alt direttamente, le altre usano data-lucide, creando incoerenza nella sorgente.
ProblemaDoppio caricamento, script di migrazione da mantenere, console.log di debug visibili in produzione, icone che potrebbero non renderizzarsi se lo script non è ancora eseguito.
SoluzioneStandardizzare tutto su Font Awesome (già caricato). Sostituire ogni data-lucide="x" con la classe FA equivalente. Rimuovere Lucide e lucide_to_fontawesome_migration.js.
A3 — Tailwind CSS 2.2.19 caricato ma quasi inutilizzato
3 MB di CSS per una manciata di utility class sulle celle della tabella modale
critico
Tailwind 2.2.19 viene caricato da CDN su index.html. Le uniche classi effettivamente usate sono poche utility su <th> e <td> nella tabella modale (px-3 py-2 text-xs font-medium text-gray-500 uppercase tracking-wider). L'intero sistema CSS custom usa variabili e classi proprietarie — Tailwind è un residuo di una vecchia fase di sviluppo.
Costo~2,9 MB di CSS inutilizzato scaricato ad ogni caricamento. Con connessioni lente (es. utenti da mobile) questo è il rallentamento principale della pagina.
SoluzioneRimuovere il tag <link> Tailwind. Aggiungere 5 righe di CSS custom per le celle tabella. Fine.
A4 — File JS monolitico da 3 371 righe
monitoraggio_p1-v9.js gestisce mappa, dati, filtri, grafici, statistiche, popup, autocomplete e molto altro
medio
Il file principale JS mescola inizializzazione mappa Leaflet, parsing CSV, logica filtri con cascata, rendering grafici SVG, gestione popup, sistema autocomplete, stili iniettati dinamicamente, e configurazione tooltip. Identificare e isolare un bug in questo contesto è costoso. Sono presenti anche blocchi // lucide.createIcons() commentati lasciati in produzione.
ProblemaDifficoltà nel debugging e nel testing. Blocchi di codice commentato (lucide.createIcons() appare 3 volte commentato). Decine di console.log con emoji visibili in produzione.
SoluzioneSeparare almeno in: map.js, data.js, filters.js, charts.js. Rimuovere codice commentato e console.log in produzione.
A5 — Variabili CSS duplicate e inconsistenti tra file
patti_pa_general.css e smart_topbar.css definiscono variabili parallele con nomi diversi per gli stessi colori
medio
--color-primary: #00264b in general.css è identico a --pst-blue: #00264b in smart_topbar.css. Stesso per molti altri colori. Questo significa che se si vuole cambiare il blu istituzionale bisogna aggiornare entrambe le variabili, in due file diversi.

Stesso problema con i colori degli stati: nel CSS sono definiti come --status-stipulato: #10b981, ma nel JS i marker sulla mappa usano 'Patto stipulato': '#8fd67d' — un verde diverso e più chiaro. La legenda mostra un colore, i pin sulla mappa ne mostrano un altro.
SoluzioneConsolidare tutto in un unico :root in general.css. Referenziare le stesse variabili nel JS via getComputedStyle(document.documentElement).getPropertyValue('--status-stipulato'). Un solo punto di verità per i colori.
A6 — Stili iniettati dinamicamente via JavaScript
addFiltersPopupStyles() crea un tag <style> nel <head> a runtime
medio
La funzione addFiltersPopupStyles() in monitoraggio_p1-v9.js inietta dinamicamente le regole CSS per .filters-popup-close e .filters-popup-reset. Questo significa che questi stili non esistono nei file CSS, sono invisibili agli strumenti di sviluppo sotto "Styles", e potrebbero confliggere con overrides futuri.
ProblemaStili invisibili nel sorgente CSS, difficili da trovare e sovrascrivere. Anti-pattern consolidato.
SoluzioneSpostare quelle 20 righe di CSS nel file patti_pa_general.css e rimuovere la funzione addFiltersPopupStyles().
A7 — Elementi nascosti nel DOM come "strato di compatibilità"
Il #hidden-filter-selects mantiene <select> invisibili come interfaccia di stato
medio
In index.html c'è un intero blocco <div id="hidden-filter-selects" style="display:none"> con 5 select, 3 checkbox, 4 span e 2 input nascosti. Questi elementi fungono da "stato condiviso" tra i vari script JS — ogni modulo legge e scrive su questi select per sincronizzarsi. È un sistema fragile: basta un getElementById che fallisce per rompere silenziosamente la logica dei filtri.
Soluzione a lungo termineCentralizzare lo stato dei filtri in un semplice oggetto JS (window.filterState = {stato:'', ambiti:'', ...}) e far sì che ogni modulo legga/scriva lì direttamente, eliminando il DOM come meccanismo di IPC tra script.
A8 — Versioning per cartella (/v9/) anziché con git
Tutti i CSS e JS vivono in sottocartelle /css/v9/ e /js/v9/
basso
Implica che esistono (o sono esistite) versioni v1–v8. Questo approccio porta a accumulare cartelle obsolete nel repo, difficoltà nel capire "quale è l'ultima versione" senza guardare il codice, e impossibilità di confrontare le diff tra versioni. Git è lo strumento corretto per questo.
SoluzioneRinominare i path in /css/ e /js/ (senza suffisso versione). Usare commit git con messaggi descrittivi e tag di release per tracciare le versioni.
03 UX & Interfaccia 7 issue
U1 — Header a 4 livelli: troppo spazio sottratto alla mappa
Su un laptop da 768px di altezza, l'header occupa ~130px (17%) prima ancora che la topbar filtri conti
critico UX
Il livello L2 ("Area Urbanistica della Rigenerazione Urbana") e il livello L3 ("Patti di Collaborazione · Report di monitoraggio") trasmettono informazioni quasi identiche e sono visibili contemporaneamente. Il livello L1 (logo Comune con slim-bar) poi sparisce allo scroll — ma solo su index.html che ha la mappa a schermo fisso. Il risultato è che la mappa — cuore dell'app — riceve pochissimo spazio verticale su schermi medi.
EffettoSu 1366×768 (laptop più diffuso in Italia): header ~140px + topbar ~48px + chip bar potenziale ~30px = la mappa inizia a 218px dal top. Solo il 60% dello schermo è mappa.
SuggerimentoFondere L2 e L3 in un unico band. Spostare il nome dell'ufficio in un tooltip o in un title del logo. Guadagno stimato: 35–45px di altezza mappa.
U2 — Statistiche e KPI nascosti in un pannello accessibile solo tramite click
Il pannello "Statistiche & Grafici" è una modale/sidebar chiusa di default — i KPI principali non sono visibili al primo colpo d'occhio
critico UX
Per una dashboard di monitoraggio, i numeri chiave (totale patti, quanti stipulati, quanti in istruttoria) dovrebbero essere immediatamente visibili. Il donut chart e le 6 KPI card sono magnificamente realizzati — ma sono nascosti. L'utente che apre l'app vede solo la mappa e deve scoprire da solo che esiste un pannello statistiche.
Opzione A (leggera)Mostrare una barra KPI sottile (5–6 numeri in orizzontale) sopra la mappa, sempre visibile. Il pannello completo rimane accessibile per i dettagli.
Opzione B (consigliata)Su desktop, aprire il pannello statistiche di default come sidebar fissa a destra. La mappa occupa il 65–70% della larghezza, le stats il restante 30–35%.
U3 — Legenda colori marker assente sulla mappa
I pin colorati sulla mappa non hanno una legenda inline visibile
medio
Un utente che apre la mappa per la prima volta vede pin di 6 colori diversi (verde, giallo, arancione, viola, rosso, grigio) senza capire cosa significano. La legenda è dentro il pannello statistiche, che è chiuso. Il controllo di layering nella mappa in basso a sinistra non chiarisce i colori degli stati.
SoluzioneAggiungere una piccola legenda floating sovrapposta alla mappa (angolo in basso a sinistra, sopra i controlli zoom), con 6 pallini colorati e le etichette degli stati. Collassabile su mobile. Massimo 120×160px.
U4 — Flusso di 2 click per accedere ai dettagli di un patto dalla mappa
Click marker → popup → click "Vedi dettagli" → side panel
medio
Il popup del marker mostra titolo, stato e un pulsante "Dettagli" che apre il side panel. Il side panel è ricco di informazioni (mini-mappa, tutti i campi, PDF). Il passaggio in due step crea attrito, specialmente su mobile dove ogni click richiede attenzione precisa sul marker.
Alternativa 1Click su marker apre direttamente il side panel (eliminare il popup intermedio). Il popup è un residuo del pattern Leaflet standard, ma il side panel è più ricco e va in quella direzione.
Alternativa 2Mantenere il popup ma renderlo più ricco (aggiungere i campi principali) così da ridurre la necessità di aprire il pannello dettagli per la maggior parte degli utenti.
U5 — Tabella dati senza ordinamento colonne né export
La modale "Vista Tabella" mostra i dati ma non permette di ordinarli per colonna né di esportarli
medio
Utenti tecnici (giornalisti, ricercatori, funzionari di altre circoscrizioni) che aprono la vista tabella vogliono tipicamente ordinare per stato o per quartiere, o scaricare i dati per analisi esterne. Nessuna delle due funzionalità è presente. I dati vengono già filtrati correttamente — manca solo l'output.
SoluzioneAggiungere click-to-sort sulle intestazioni (<th>) con frecce di direzione. Aggiungere un pulsante "Scarica CSV" che usa Blob + URL.createObjectURL per generare il file lato client dai dati già in memoria — nessuna dipendenza server.
U6 — Loading state della mappa non comunicato
Prima che i dati CSV arrivino da GitHub, la mappa è vuota e il contatore risultati mostra "—"
medio
Il fetch del CSV da GitHub Raw può richiedere 1–3 secondi su connessioni lente. In quel lasso di tempo la mappa è vuota di marker, i KPI mostrano tutti "0", e il contatore risultati mostra "—". Un utente inesperto potrebbe pensare che l'app sia rotta o che non ci siano dati.
SoluzioneAggiungere uno stato di caricamento visibile: spinner o skeleton sui KPI card, overlay sulla mappa con messaggio "Caricamento dati in corso…", e gestione esplicita dell'errore (se il fetch fallisce, mostrare un messaggio chiaro con il tasto "Riprova").
U7 — "Attiva un Patto" è la pagina più utile per i cittadini, ma è al 4° posto nella nav
L'ordine dei tab non rispecchia l'importanza dei contenuti per il pubblico target
basso
Il tab "Attiva un Patto!" — che contiene le istruzioni operative, la guida passo-passo e i contatti per procedere — è il quarto link nella navigazione. Per un cittadino che non conosce il sistema, questa è la pagina più importante. Dashboard e Riferimenti Normativi sono più utili per funzionari che già conoscono il sistema.
SuggerimentoValutare di riordinare i tab per audience: Dashboard → Attiva un Patto! → FAQ → Modulistica → Normativi → Laboratori → About → Info. In alternativa, evidenziare "Attiva un Patto!" con uno stile diverso (es. pulsante accentato) per renderlo visivamente prioritario.
04 Performance 5 issue
P1 — Peso complessivo CSS stimato: ~3,4 MB (di cui ~97% inutilizzato)
Tailwind 2.2.19 CDN: ~2,9 MB · Font Awesome: ~400 KB · CSS custom: ~110 KB
critico
Il CSS totale scaricato su index.html è dominato da Tailwind (non purgato) e Font Awesome (tutte le 2 000+ icone, mentre ne vengono usate circa 30). Su una connessione 3G tipica (1,5 Mbps), solo il CSS richiede ~18 secondi di download.
FonteDimensioneUtilizzo effettivoAzione
Tailwind 2.2.19 CDN~2,9 MB<1%Rimuovere
Font Awesome 6.5.1~400 KB~1,5%Subset o self-host
CSS custom (7 file)~110 KB~70%Consolidare
P2 — Nessuna strategia di caching per i dati CSV
I dati vengono rifetchati da GitHub Raw ad ogni caricamento pagina
medio
Il CSV viene scaricato con un semplice fetch() senza header di cache. Ogni apertura della pagina recupera l'intero file. Se GitHub è temporaneamente irraggiungibile, l'app si rompe completamente — nessun fallback.
RischioDipendenza da GitHub Raw come CDN dati. Un downtime di GitHub (raro ma possibile) rende l'app inutilizzabile.
SoluzioneAggiungere Cache-Control via Service Worker o usare localStorage come cache con TTL (es. 1 ora). In alternativa, ospitare il CSV sullo stesso dominio dell'app.
P3 — 14 script JS caricati sequenzialmente in fondo al body
Nessun attributo defer o async sugli script non critici
medio
Gli script sono caricati nel corretto ordine di dipendenza, ma senza ottimizzazioni di caricamento. Script come back_to_top.js, mobile-nav.js, slim-header.js potrebbero essere caricati con defer senza impatti.
Soluzione rapidaAggiungere defer ai 5–6 script non dipendenti dal DOM iniziale (back_to_top.js, mobile-nav.js, slim-header.js, url-routing.js). Il browser li scarica in parallelo e li esegue solo dopo il parsing HTML.
P4 — Immagini del footer non ottimizzate né lazy-loaded
logo_pa_01.png, s_rosalia_white.png e altri asset del footer si caricano subito
basso
Il footer compare solo dopo lo scroll. Le immagini che contiene vengono però scaricate immediatamente. Su index.html (dove la mappa è a schermo pieno e il footer è fuori dal viewport) questo è spreco puro.
SoluzioneAggiungere loading="lazy" alle immagini del footer. Due parole di HTML, zero impatto funzionale, caricamento differito finché l'utente non scrolla.
P5 — Decine di console.log con emoji in produzione
"✅ chartFilterState inizializzato", "🔍— Ricerca intelligente unificata inizializzata…" e molti altri
basso
I file JS di produzione contengono decine di console.log, console.warn e console.error lasciati dal debugging. Non impattano significativamente le performance, ma inquinano la console del browser (visibile a chiunque apra i DevTools), rivelano dettagli implementativi, e creano un'impressione di scarsa pulizia del codice.
SoluzioneCreare un wrapper logger che attiva i log solo se location.hostname === 'localhost' oppure se è impostato un flag DEBUG=true in localStorage. Sostituire tutti i console.log con logger.log.
05 Accessibilità 4 issue
AC1 — Focus trap mancante nei modali
I 3 modali (#pattoModal, #tableModal, #tabsModal) hanno role="dialog" ma nessun trap del focus
critico a11y
Quando un modale è aperto, premere Tab dovrebbe mantenere il focus all'interno del modale (focus trap). Senza di questo, un utente che naviga da tastiera o con screen reader uscirà dal modale e interagirà con elementi sottostanti, che è un comportamento confusionario e contro le WCAG 2.1 AA (criterio 2.1.2).
SoluzioneImplementare focus trap: all'apertura, salvare l'elemento che aveva il focus; intercettare Tab/Shift+Tab per ciclare solo tra elementi focusable dentro il modale; a chiusura, ripristinare il focus sull'elemento originale. ~30 righe di JS vanilla, riusabile per tutti i modali.
AC2 — FAQ realizzate con <div> invece di <details>/<summary>
I 8 item FAQ in faq.html usano .step-item statici, non espandibili e non accessibili da tastiera
medio
Gli item FAQ sono attualmente tutti aperti simultaneamente in una lista numerata statica. Non c'è interazione, non c'è accordion. Un utente che naviga da mobile deve scorrere 8 risposte complete. L'elemento HTML nativo <details>/<summary> è accessibile, espandibile, e non richiede JavaScript.
SoluzioneWrappare ogni FAQ in <details><summary>Domanda</summary>Risposta...</details>. Zero JavaScript richiesto. Accessibilità nativa. Stile personalizzabile con CSS su details[open].
AC3 — Struttura heading non corretta in attiva.html
I capitoli dei contenuti usano <h1> annidati dentro il <main> che già ha un <h3> come titolo sezione
medio
In attiva.html, i capitoli del regolamento usano <h1> per i titoli interni (es. "L'amministrazione condivisa dei beni comuni") mentre il titolo di pagina visibile è marcato come <h3>. Un screen reader annuncierà un <h1> dentro un <h3> come struttura incoerente, rendendo la navigazione per heading confusionaria.
SoluzioneApplicare una gerarchia corretta: <h1> per il titolo pagina (che esiste già nell'header istituzionale), <h2> per i capitoli, <h3> per le sezioni interne, <h4> per le sottosezioni.
AC4 — Link nel footer puntano a # (Privacy, Cookie, Accessibilità)
Tre link nel footer bottom indicano pagine inesistenti o non ancora create
basso
I link Privacy policy, Cookie e Accessibilità in ogni footer puntano a href="#". Per un sito di un ente pubblico, la dichiarazione di accessibilità è obbligatoria per legge (D.Lgs. 106/2018 che recepisce la Direttiva UE 2016/2102). Anche la privacy policy è un obbligo GDPR.
Azione richiestaCreare le pagine privacy.html, cookie.html, accessibilita.html e aggiornare gli href. La dichiarazione di accessibilità può usare il template AGID disponibile su form.agid.gov.it/accessibilita.
06 Manutenibilità 4 issue
M1 — CSS eccessivamente frammentato in 7 file (3 400+ righe nel principale)
patti_pa_general.css è 3 404 righe; patti_pa_media.css è solo 50 righe
medio
La frammentazione non segue una logica coerente: patti_pa_media.css ha 50 righe di media query che potrebbero stare alla fine di general.css, mentre smart_topbar.css è correttamente separato perché è un componente autonomo. Il file da 3 404 righe non ha un sommario/indice — trovare una regola specifica richiede Ctrl+F.
SuggerimentoConsolidare patti_pa_media.css e patti_pa_footer_media.css nei rispettivi file principali (le media query stanno bene alla fine del file che descrivono). Aggiungere un indice commentato all'inizio di general.css con i numeri di riga delle sezioni principali.
M2 — Codice commentato e commenti "TODO" abbandonati nel sorgente
Blocchi lucide.createIcons() commentati 3 volte, comment block vuoti, vecchie versioni di funzioni
medio
In monitoraggio_p1-v9.js: // lucide.createIcons() compare 3 volte come codice commentato. In index.html: il commento <!-- CSS --> <!-- --> sulla riga 16 è un artefatto vuoto. In tab-modal.js e side-panel.js ci sono probabilmente altri residui (non letti per brevità). Il codice commentato crea rumore cognitivo e fa sorgere dubbi ("perché è commentato? va decommentato?").
AzioneFare un passaggio di pulizia: eliminare tutto il codice commentato che non serve, rimuovere i console.log diagnostici, togliere i commenti TODO risolti. Git preserverà la storia del codice rimosso — non serve tenerlo inline.
M3 — window.* usato come sistema di state management
window.allData, window.filteredData, window.chartFilterState, window.getProponenteFilter…
basso
Lo stato dell'app è distribuito su variabili globali, hidden select nel DOM, window.* properties, e lo URL. Questo rende il debugging difficile ("dove è settato questo valore?") e fragile (qualsiasi script esterno potrebbe sovrascrivere window.allData).
Obiettivo a lungo termineCentralizzare tutto in un unico oggetto AppState con getter/setter espliciti. Non è urgente — ma ogni nuova feature dovrebbe usare questo pattern per ridurre gradualmente il debito.
M4 — Nessun README tecnico per gli sviluppatori
Il README.md spiega l'app agli utenti, non agli sviluppatori
basso
Il README descrive cosa sono i patti e come usare la dashboard — ottimo per gli utenti. Ma non documenta: come aggiornare il CSV dei dati, quali campi del CSV sono attesi, come aggiungere una nuova pagina, cosa fa ciascun file JS, come testare in locale.
SuggerimentoAggiungere una sezione "Sviluppo" al README con: struttura file, come fare girare in locale (es. npx serve .), come aggiornare i dati, formato CSV atteso con descrizione campi.
07 Contenuto & Dati 4 issue
C1 — Colori degli stati inconsistenti tra CSS e JavaScript
La legenda mostra #10b981 (verde intenso) ma i marker sulla mappa usano #8fd67d (verde più chiaro)
medio
In patti_pa_general.css: --status-stipulato: #10b981. In monitoraggio_p1-v9.js (oggetto statusColors): 'Patto stipulato': '#8fd67d'. Sono due verdi visibilmente diversi. Un utente che guarda la legenda nel pannello statistiche vedrà un verde, ma i marker sulla mappa avranno un altro verde. Stessa incongruenza per gli altri stati.
SoluzioneCome indicato in A5: usare un unico punto di verità. Nel JS, leggere i colori dalle variabili CSS: getComputedStyle(document.documentElement).getPropertyValue('--status-stipulato').trim(). Eliminare l'oggetto statusColors hardcoded.
C2 — Typo "Rigenerazione Urbana" ripetuto su tutte le pagine
Manca la 'r' — compare nel footer di ogni pagina e nell'intestazione dell'ufficio
medio
Il testo "Area Urbanistica della Rigenerazione Urbana" (invece di "Urbana") appare nel footer di tutte e 8 le pagine, due volte per footer (nella descrizione dell'ufficio e in quella dell'area). Essendo duplicato in ogni file, si è propagato ovunque. Questo è esattamente il tipo di errore che si eliminerebbe con un componente footer centralizzato (cfr. A1).
Fix immediatoCerca-e-sostituisci "Rigenerazione Urbana" → "Rigenerazione Urbana" in tutti i file HTML. Fix definitivo: centralizzare il footer (A1) così non si ripete.
C3 — Copyright "© 2025" e riferimento "1624-2025" non aggiornati
Il footer mostra l'anno scorso; il sottotitolo del logo S. Rosalia riferisce il 2025
basso
Il copyright è statico nel sorgente HTML: © 2025. Siamo nel 2026. Il riferimento "400+1 Festino Santa Rosalia 1624-2025" si riferisce a un evento del 2025 ormai passato — probabilmente il logo andrebbe rimosso o aggiornato.
Soluzione copyrightRendere l'anno dinamico: <span id="footer-year"></span> + document.getElementById('footer-year').textContent = new Date().getFullYear() nel footer JS. Zero manutenzione futura.
C4 — footer-loghi-placeholder vuoto su tutte le pagine
Un <div class="footer-loghi-placeholder"></div> è presente ma privo di contenuto
basso
La struttura del footer prevede una sezione per loghi/marchi partner nell'area superiore destra, ma è rimasta vuota. Se i loghi non ci sono, il <div> vuoto andrebbe rimosso per non occupare spazio inutile nel layout e nel DOM.
AzioneDecidere: inserire i loghi previsti (OpenDataSicilia? partner istituzionali?) oppure rimuovere completamente il div placeholder e le sue regole CSS associate.
08 Piano d'azione prioritario

Organizzato in 3 sprint progressivi. Ogni sprint è indipendente e produce valore anche se gli altri non vengono eseguiti.
